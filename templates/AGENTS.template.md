# Agent Instructions

These are common instructions for agents across all scenarios.

## General Guidelines

- Never use the em dash.
Use a plain dash instead.
- When writing a commit message, PR description, or comment, NEVER auto-add your agent name as a co-author.
- Never manually modify CHANGELOG.md files or any files that are marked as autogenerated.
- When writing or substantially editing long markdown files, put each full sentence on its own line.
Preserve normal markdown structure, but avoid wrapping multiple sentences onto one physical line.
- When doing bug fixes, always start by reproducing the bug in an end-to-end (E2E) setting, as closely aligned with how an end user would experience it as possible.
This makes sure you find the real problem so your fix will actually solve it.
- When end-to-end testing a product, be picky about the UI you see and be obsessed with pixel perfection.
If something clearly looks off, even if it is not directly related to what you are doing, first create a bug, then a separate branch, try to get it fixed, and relate the existing work to this bug.
- Apply the same high standard of engineering excellence to lint, test failures, and test flakiness.
If you see one, even if it is not caused by what you are working on right now, still get it fixed.

## Opinionated

When you are working on something that would benefit from being informed by my viewpoints, read opinions.md to understand them.

## Voice Profile

When you are talking or posting on behalf of me, using my identity, read ./voice.md for tone.
